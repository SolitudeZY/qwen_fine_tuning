# 基于 Qwen-VL 的施工现场安全合规检测模型微调实践

## 一、 业务背景与技术选型痛点

*   **业务场景**：我们需要对施工现场（如基坑、边坡、便道、盖梁作业平台等）的“临边防护（围栏）”进行安全合规检测，不仅要判断是否违规，还要精确定位违规位置（如围栏断口、倒伏、未合围）。
*   **传统视觉方案的局限**：传统的目标检测（如 YOLO）只能识别“这里有个围栏”，但很难理解“围栏是否合围”、“平台间隙是否安全”这种**强逻辑关联**的复杂场景。
*   **VLM 基座模型的幻觉（Hallucination）**：
    *   直接使用未经微调的大模型，它往往会产生严重的“幻觉”。
    *   **典型翻车案例**：模型在推理时明明说“这里有围栏断口”，但最后输出的 JSON 结论却是“合规（False）”。或者面对极小的断口，因为分辨率和领域知识不足，它会“视而不见”。
*   **解决方案**：项目采用两条并行微调路线：以 **Qwen2.5-VL-7B-Instruct** 作为通用临边监测主线，以参数量更轻量（2B）的 **Qwen3-VL-2B-Instruct** 作为围栏精细化专项线，均通过 **QLoRA** 技术进行领域内垂直微调。

---

## 二、 数据飞轮——高质量微调数据的构建

在微调中，我们发现**数据的质量远比数量重要**。我们构建数据飞轮经历了以下三个关键步骤：

### 1. 标签体系重构与精细化标注
*   **问题**：最初的数据集中，很多图片只有粗略的“违规”标签，甚至混入了根本没有围栏的“安全帽/反光衣”图片，导致模型被严重污染。
*   **动作**：我们将违规类别细化为具体的物理现象（如 `围栏断口`、`围栏倒伏`、`临边无防护`、`踢脚板缺失`）。使用 LabelMe 重新对这 100 多张图片进行精准的边界框（BBox）标注。

### 2. Teacher Model 辅助推理生成（数据合成）
*   大模型不仅需要坐标，还需要”思考过程”来生成高质量标注。
*   我们编写了脚本，将 LabelMe 的绝对像素坐标转换为 Qwen-VL 认识的 **`0-1000` 千分比相对坐标**。
*   通过调用阿里最强的 **Qwen-VL-Max API** 作为 Teacher Model，把真实坐标喂给它，让它反推生成一段”专家级”的安全检查推理过程（Chain-of-Thought），最后组装成标准的 `JSONL` 多轮对话格式。
*   **关键清洗步骤**：Teacher Model 生成的 CoT 推理文本在后续数据清洗阶段（`prepare_train_v2_data.py`）会被剥离，最终训练目标统一为**纯 JSON 输出**，避免格式污染和输出抖动。

### 3. 克服数据倾斜（Data Imbalance）
*   **踩坑**：如果只用违规图片训练，模型会变成一个”只要看到工地就报违规”的机器（假阳性极高）。
*   **动作**：我们通过爬虫和数据筛选，专门准备了约 35 张**完美合规**的围栏图片（负样本），生成了没有任何违规框的空 JSON 标注（`violation_boxes: []`），混合进训练集。最终平衡后的训练集共 135 条样本（含合规与违规混合），这极大降低了模型的误报率。

## 三、 LoRA 原理与微调超参数的深度调优

### 1. 为什么选择 QLoRA？
*   全量微调 2B 的多模态模型成本太高。我们使用了 **QLoRA (Quantized Low-Rank Adaptation)**。
*   **原理**：冻结预训练好的 Qwen3-VL 基座模型权重，并将其量化为 4-bit（极大降低显存占用，只需单卡即可训练），同时在注意力层（Attention）旁挂载小型的、可训练的“旁路矩阵”（LoRA Adapter）。我们只训练这不到 3% 的新增参数，让模型“低成本学会新知识”。

### 2. 超参数调优演进（核心干货）
在测试中，我们发现模型经常发生“长文本被强行截断（导致 Parse Error）”或“没看清远处小围栏缺口”的问题。我们针对性地修改了启动脚本 `train_fences.sh` 中的超参数：

| 超参数 | 初版设置 | 最终调优版本 | 调优动机与原理 |
| :--- | :--- | :--- | :--- |
| **`MAX_PIXELS` (分辨率)** | `602112` | **`1003520`** | **极其关键！** 建筑工地多为无人机高空俯拍，围栏像素占比极小。将视觉 tokens 的最大像素上限提升到 100W+（约 1000x1000 分辨率），彻底治好了模型的“近视眼”，大幅提升了小目标的检测率。 |
| **`EPOCHS` (训练轮数)** | `5` | **`15`** | 对于 VLM 而言，不仅要学会输出格式，还要将视觉特征与特定文本逻辑绑定。5 轮时模型偶尔会出现”推理与结论不一致”的幻觉，增加到 15 轮让模型把领域知识吃得更透（Fit better）。 |
| **`LR` (学习率)** | `1e-4` | **`5e-5`** | 随着 Epoch 的增加，为了防止模型步子迈得太大导致灾难性遗忘（Catastrophic Forgetting）或过拟合，我们降低了学习率，让模型在更精细的解空间中收敛。 |
| **`BATCH_SIZE` / `GRAD_ACCUM`** | `2` / `2` | **`1` / `4`** | 因为提高了分辨率，单卡显存压力激增。我们将单次 Batch Size 降为 1 防止 OOM，同时将梯度累加（Gradient Accumulation）提升至 4，以维持等效的全局 Batch Size 保证梯度更新的稳定性。 |
| **`LORA_RANK` (r)** | `64` | **`32`** | 初版（`train.sh`）使用 rank=64, alpha=128，参数量较大。围栏专项版降至 rank=32, alpha=64，在保持足够表达能力的同时减少过拟合风险。多模态任务特征复杂，rank 仍高于常规文本微调（通常 8 或 16）。 |

## 四、 测试评估与业务收益

### 1. 评估工程（Evaluation Pipeline）
*   我们开发了一套全自动的测试脚本 `test_fence_violations.py`。
*   **兜底解析器**：由于小模型（2B）偶尔不按套路出牌，我们编写了强大的正则表达式兜底逻辑。即使模型把 `violation_boxes` 散落在文本各处，或者使用了单引号，脚本也能准确抓取并还原 JSON。

### 2. 测试结果对比（Base vs LoRA）
*   **基座模型（无 LoRA）**：在违规数据集上，基座模型面对复杂的 JSON 输出指令经常崩溃（Parse Error），且容易漏检（Recall 极低），无法满足业务需求。
*   **微调模型（带 LoRA）**：在跑完 15 个 Epoch 后，微调模型不仅能 100% 遵循严格的 JSON 格式输出，还能精准锁定坐标，并自动保存画框的可视化图片。输出为结构化纯 JSON，便于下游安全巡检系统直接接入。

## 五、 总结与下一步展望

*   **总结**：VLM（视觉大模型）在工业安防领域的落地，核心在于 **“高质量的专家标注 + Teacher Model数据合成 + 针对性的高分辨率 QLoRA 微调”**。
*   **展望**：未来我们可以引入更多负样本，或者将更多类别的隐患（如泥浆池无防护、安全带未系）加入标签体系，不断扩大这套微调范式的业务边界。
